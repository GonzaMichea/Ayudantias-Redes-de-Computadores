Parte 1: Conceptos Fundamentales

1. Protocolos de Aplicación y Encapsulación

¿Qué es un Protocolo de Aplicación?

Un Protocolo de Aplicación es un conjunto estructurado de reglas que determinan cómo las aplicaciones en diferentes computadores se comunican entre sí. Define:

  • El tipo de mensajes que se intercambian (ej. Peticiones/Requests y Respuestas/Responses).
  • La sintaxis de esos mensajes (qué campos tienen y cómo están organizados).
  • La semántica (el significado de la información en esos campos).
  • Las reglas para saber cuándo y cómo enviar los mensajes.

El Flujo de Bits en la Encapsulación

Cuando envías un mensaje (ej. un email), este no viaja mágicamente por el cable. Tiene que descender por las capas de red, sufriendo un proceso llamado encapsulación, donde cada capa envuelve al mensaje con un encabezado (Header) nuevo.

Data Link Header Añade dir. MAC (Trama)
IP Header Añade dir. IP (Paquete)
Transport Header Añade Puerto TCP/UDP (Segmento)
Application Data El mensaje original (HTTP, SMTP, etc.)

Del mensaje a los bits: Una vez que el mensaje original tiene todos estos encabezados (MAC + IP + Puerto + Datos), la tarjeta de red (Capa Física) toma este "paquete" gigante de ceros y unos y lo convierte en señales eléctricas, pulsos de luz o radiofrecuencias (Wi-Fi) para enviarlo por el medio físico. Al llegar al destino, ocurre el proceso inverso (Desencapsulación).

2. Puertos, Sockets y la IANA

¿Qué es la IANA?

La IANA (Internet Assigned Numbers Authority) es la entidad global encargada de coordinar los identificadores únicos en Internet, como las direcciones IP y, muy importante, los números de puerto. Los clasifica así:

  • 0 - 1023: Puertos Conocidos (Requieren permisos de root. Ej: 80, 443).
  • 1024 - 49151: Puertos Registrados (Para software específico).
  • 49152 - 65535: Puertos Dinámicos (Usados temporalmente por clientes).

Socket vs Puerto

Es común confundirlos, pero son cosas distintas:

  • El Puerto: Es solo un identificador lógico numérico (como el número de un departamento en un edificio). Sirve para que el Sistema Operativo sepa a qué programa entregar el mensaje.
  • El Socket: Es la estructura de software (interfaz de programación) completa que un programa usa para conectarse a la red. Se define por la combinación de: [Dirección IP + Puerto + Protocolo de Transporte (TCP/UDP)]. Es la "puerta física" real de comunicación abierta por el programa.

3. Conexiones Remotas: TELNET vs SSH

En la capa de aplicación existen protocolos específicos para administrar computadores y servidores de forma remota a través de una terminal o consola de comandos.

TELNET (Puerto 23)

Es uno de los protocolos más antiguos de Internet. Permite conectar la terminal de tu computador a la de un servidor remoto.

⚠️ El Problema: TELNET envía ABSOLUTAMENTE TODO en texto plano (sin cifrar). Si escribes tu contraseña de administrador, viaja por el cable como texto legible. Cualquiera haciendo sniffing en la red puede robar tu acceso al instante. Hoy en día se considera obsoleto e inseguro.

SSH (Puerto 22)

SSH (Secure Shell) nació como el reemplazo seguro de TELNET. Cumple exactamente la misma función de acceso a consola remota, pero con seguridad integrada.

✅ La Solución: SSH utiliza criptografía fuerte. Toda la comunicación, desde la autenticación (usuario/contraseña) hasta los comandos que ejecutas, viaja cifrada. Si un atacante intercepta el tráfico, solo verá caracteres incomprensibles.

4. La Web, HTTP y Certificados X509

El protocolo HTTP (Puerto 80) rige la Web basándose en un modelo Cliente-Servidor. Es un protocolo Stateless (sin estado), lo que significa que el servidor no recuerda peticiones pasadas del mismo cliente por defecto (por eso se inventaron las cookies).

HTTPS (Puerto 443) y Certificados X509

Igual que el problema de Telnet, HTTP envía los datos en texto plano. HTTPS añade seguridad cifrando la conexión. Pero, ¿cómo sabes que el servidor al que te conectas es realmente tu banco y no un impostor? Para eso está el Certificado X509.

El servidor envía este certificado emitido por una CA (Autoridad de Certificación) confiable. El navegador verifica la firma de la CA; si no coincide, expiró o el dominio es distinto, el navegador te bloqueará con una alerta de seguridad ("Connection is not secure").

5. Escalabilidad: Balanceo de Cargas y P2P

Balanceo de Cargas

Para evitar que un servidor colapse por demasiadas peticiones, se distribuye el tráfico:

  • Rotación DNS (Round Robin): El servidor DNS da una IP distinta a cada cliente para repartirlos entre varios servidores.
  • HTTP Redirect: El servidor principal responde al cliente con un código indicándole que vaya a otro servidor menos cargado.
  • HAProxy: Un software especializado (Proxy) que se pone delante y distribuye el tráfico inteligentemente a servidores internos ocultos.

Modelo Peer-to-Peer (P2P)

Redes descentralizadas. Todos los nodos actúan como clientes (descargan) y servidores (suben) simultáneamente (Ej: BitTorrent).

✓ Ventajas:

Escalabilidad natural: a más usuarios, más ancho de banda total. Alta tolerancia a fallas centrales.

✗ Desventajas:

Difícil controlar la distribución de malware, la piratería o asegurar que un archivo siempre esté disponible.

Parte 2: Preguntas de Análisis Conceptuales

Estas preguntas provienen directamente de la Ayudantía 3 de la asignatura. Haz clic en cada pregunta para revelar el análisis detallado.

Porque el puerto es simplemente un número asignado (el identificador lógico del servicio, ej. el puerto 80 para HTTP). En cambio, el socket es el objeto de software completo (la interfaz de programación o API) instanciado por el Sistema Operativo que hace la conexión real a la red, y está formado por la dirección IP de la máquina, el número de puerto y el protocolo a usar (TCP o UDP).

Porque HTTP transmite todos los datos (incluyendo contraseñas, correos y tarjetas de crédito) en texto plano. Cualquier atacante intermediario (ej. en un WiFi público) puede leer la información usando un software de captura de paquetes (Sniffer). HTTPS cifra la comunicación mediante SSL/TLS, haciendo que los datos interceptados sean ilegibles.

Ventajas: Altamente escalable, ya que la capacidad de distribución de la red crece orgánicamente con cada nuevo nodo que se une. No existe un punto único de falla (si se cae un usuario, el archivo sigue disponible en otros).

Desventajas: No hay un control central, lo que fomenta la distribución de malware o contenido ilegal. La disponibilidad de un recurso no está garantizada (si los usuarios que tienen un archivo se desconectan, el archivo se pierde).

Para enviar mensajes de texto, generalmente no. UDP no garantiza la entrega de paquetes ni el orden de estos, por lo que un mensaje importante podría simplemente perderse en la red y la aplicación nunca se enteraría. Para mensajes, textos y fotos se prefiere TCP (o protocolos que añaden fiabilidad). Excepción: Si hablamos de las llamadas de voz o video dentro de esa misma app, ahí SÍ se prefiere UDP, ya que importa más la velocidad en tiempo real que recuperar un fotograma perdido.

Permiten multiplexar las comunicaciones. Mientras la dirección IP permite que un paquete llegue a la máquina correcta, el Puerto indica a qué programa o servicio específico dentro de esa máquina debe entregarse el mensaje (ej. Puerto 80 va al servidor web Apache, Puerto 22 va al servicio SSH).

A través de un Certificado Digital (X509). Cuando nos conectamos, el servidor envía este certificado que funciona como su "cédula de identidad". Este documento ha sido validado criptográficamente y firmado digitalmente por una entidad de confianza global llamada CA (Autoridad de Certificación). El navegador del usuario revisa esta firma; si es válida, sabe que el dominio es genuino y no un impostor.

Caso Práctico 1: Elección de Socket

Diseño de una Aplicación de Telemetría

Estás diseñando el software para un dron autónomo. El dron envía sensores de su inclinación y temperatura cada 10 milisegundos hacia un servidor en tierra para dibujarlos en una gráfica en vivo. Además, ocasionalmente el servidor en tierra le envía al dron comandos críticos como "Aterrizaje de Emergencia".

Pregunta: ¿Qué tipo de protocolo de transporte (TCP o UDP) usarías para enviar la telemetría, y cuál usarías para los comandos críticos? Justifica tu respuesta.

Resolución del Caso

1. Para la Telemetría (Sensor de datos): Usar UDP.

Justificación: El dron manda datos 100 veces por segundo. Si se pierde un paquete con la temperatura de un milisegundo específico, no sirve de nada que el protocolo lo retransmita medio segundo después, porque ya tendríamos lecturas más nuevas y actualizadas. UDP es veloz, ligero y perfecto para flujos de datos en tiempo real donde la pérdida ocasional es tolerable.

2. Para los Comandos Críticos ("Aterrizaje"): Usar TCP.

Justificación: Si el servidor manda una orden de aterrizar y el paquete se pierde, el dron se estrellaría. Para este canal se necesita TCP (Transmisión Controlada), ya que garantiza que el mensaje llegará correctamente. Si TCP detecta que el paquete se perdió en el aire, lo volverá a retransmitir automáticamente.

Caso Práctico 2: Balanceo de Carga

La caída del Servidor de Matrículas

La universidad ha implementado un sistema donde un servidor web principal tiene implementado un mecanismo de HTTP Redirect. Cuando un alumno entra, este servidor le responde enviando un código HTTP 302 que dice: "Ve al servidor B (IP: 200.x.x.x)".

Pregunta: El día de toma de ramos, entran 10.000 alumnos al mismo tiempo. Los servidores de apoyo B y C tienen capacidad de sobra, pero el sistema completo se cae. Basado en la arquitectura, ¿qué cuello de botella falló y cómo se solucionaría?

Resolución del Caso

El cuello de botella fue el propio servidor principal de redirección.

Análisis: El balanceo por HTTP Redirect tiene una gran debilidad: para que un cliente sea redirigido a un servidor desocupado, primero tiene que abrir una conexión TCP completa con el servidor principal, el cual debe procesar la solicitud a nivel de aplicación web y emitir una respuesta. Si 10.000 alumnos llegan de golpe, el servidor principal agota todos sus sockets disponibles y sus recursos de CPU solo tratando de redirigir, colapsando y dejando a los servidores B y C inalcanzables.

Solución: Reemplazar la redirección HTTP en la capa de aplicación por una estrategia a nivel de red como HAProxy (que distribuye el tráfico velozmente antes de procesarlo a fondo) o utilizar Rotación DNS para que la repartición de IPs se haga antes de que los alumnos toquen siquiera los servidores de la universidad.

Caso Práctico 3: P2P y BitTorrent

Roles en una Red Descentralizada

En una arquitectura P2P usando BitTorrent, un usuario desea descargar el último sistema operativo Linux (archivo `.torrent`). Analizando las diapositivas de la asignatura:

Pregunta: ¿Qué rol cumple el "Tracker" dentro de la red BitTorrent? ¿El tracker almacena el archivo Linux que el usuario quiere descargar?

Resolución del Caso

1. ¿El tracker almacena el archivo?
NO. El Tracker nunca almacena el archivo (el sistema Linux). La característica central del P2P es que el archivo lo tienen los propios usuarios (Nodos/Peers).

2. ¿Qué rol cumple entonces?
El Tracker actúa como un "director de tráfico" o una libreta de direcciones. Cuando abres el archivo .torrent, tu cliente contacta al Tracker. El Tracker le responde diciendo: "No tengo el archivo, pero aquí tienes una lista de 500 IPs de otros usuarios en el mundo que sí lo están compartiendo en este instante". Tras obtener esa lista, tu computador se conecta directamente a esos usuarios (P2P) para empezar a intercambiar los trozos de archivo.
